title: "Security Infrastructure"
---

The platform is built on enterprise-grade infrastructure designed to protect data at every stage of the call lifecycle — from the moment a call connects to the moment the recording is stored.

### **Architecture**

The platform consists of three core components: a backend API (Node.js/Express) running on AWS EC2 with PostgreSQL on AWS RDS for all persistent data, a React-based dashboard served via Nginx, and a real-time voice agent engine (Python) running on LiveKit. All components are containerised with Docker and sit behind Caddy for reverse proxy and TLS termination.

This separation means that the dashboard, the API, and the voice processing layer are isolated from each other. A compromise of one component does not automatically expose the others.

### **Encryption**

All data in transit between your browser and the platform is encrypted via TLS (HTTPS). API calls between platform components use the same standard. Call recordings stored in AWS S3 benefit from server-side encryption at rest. Database connections to PostgreSQL on AWS RDS are encrypted in transit. Payment data is handled exclusively by Stripe and never stored on [RapidCall.ai](http://RapidCall.ai) servers.

### **Authentication & Workspace Isolation**

Authentication is JWT-based, with session tokens scoped to individual workspaces. Each workspace is fully isolated — its agents, phone numbers, call history, recordings, billing, and team members are completely separate from every other workspace on the platform. There is no data overlap between workspaces, even when they belong to the same parent account. This is by design: agencies managing multiple clients can give each client their own workspace with zero risk of cross-contamination.

### **Voice Processing**

Real-time voice processing runs on LiveKit, an open-source WebRTC framework. Voice data is streamed in real time between the caller, the speech recognition engine (Deepgram), the language model, and the text-to-speech engine. This streaming architecture means audio is processed as it arrives — it is not stored in intermediate buffers or temporary files during the conversation. Once the call ends, the final recording (if enabled) is written to S3, and the transcript is persisted to the database.

### **Third-Party Provider Security**

The platform’s voice stack relies on leading infrastructure providers, each with their own security posture:

| **Provider** | **Role** |
| :-- | :-- |
| AWS (EC2, RDS, S3) | Compute, database, and object storage. SOC 2, ISO 27001, and GDPR-compliant infrastructure. |
| Twilio | Telephony and SIP trunking. SOC 2 Type II certified. GDPR Data Processing Addendum available. |
| LiveKit | Real-time WebRTC voice processing. Open-source, SOC 2 Type II certified (Cloud). |
| Deepgram | Speech-to-text. SOC 2 Type II certified. Processes audio in real time without persistent storage. |
| Stripe | Payment processing. PCI DSS Level 1 certified. No payment data touches [RapidCall.ai](http://RapidCall.ai) servers. |
| ElevenLabs / Cartesia | Text-to-speech. Audio is generated in real time and streamed to the caller. |
| Anthropic / OpenAI / Google | Language model inference. Inputs are processed for response generation. |

\

We select providers based on a combination of performance, reliability, and security posture. The infrastructure underpinning your calls — from AWS and Twilio to LiveKit and Deepgram — is the same enterprise-grade stack used by organisations that operate under strict regulatory requirements. We build on top of it so that the security baseline is high before any platform-specific measures are applied.

### **Account Security Best Practices**

A few practical steps to keep your account secure:

**Use strong, unique credentials. **Your login credentials protect access to your agents, call recordings, contact data, and billing. Treat them accordingly.

**Use team roles instead of shared logins. **The platform supports Owner and Member roles. Add team members with individual accounts rather than sharing a single set of credentials. This gives you visibility into who accessed what and makes it easy to revoke access when someone leaves the team.

**Rotate API keys periodically. **If you are using the API for custom integrations, rotate your keys on a regular schedule. Revoke any keys that are no longer in use.

**Review your workspace access. **Periodically check who has access to each workspace, particularly if you manage multiple workspaces for different clients. Remove access for anyone who no longer needs it.
