---
title: "How Outbound Campaigns Work"
---

An outbound campaign is any workflow where your AI agent systematically calls a list of contacts rather than placing one-off calls manually.

RapidCall does not use a built-in campaign manager with a single “start campaign” button. Instead, outbound campaigns are driven through external workflows that trigger calls using the RapidCall API.

This is intentional. It gives you full control over how campaigns run, including timing, sequencing, retry logic, concurrency, and how data moves between systems.

In most cases, an outbound campaign has four core components.

### 1. Contact list

The contact list is the source of truth for who gets called and what information the agent receives when the call begins.

This can be:

- a Google Sheet,
- an Excel file,
- a CRM export,
- a database table,
- or any structured data source.

At minimum, it should contain the contact’s phone number and any other data the agent needs to personalise the call, such as:

- name,
- email,
- account number,
- appointment time,
- lead type,
- notes,
- or other dynamic variables.

### 2. Workflow engine

The workflow engine is the orchestration layer that reads contacts from the list and tells RapidCall when to place calls.

Common options include:

- [**Make.com**](http://Make.com)
- **n8n**
- **Zapier**
- or a custom-built script

This layer controls:

- when calls are triggered,
- how many calls run at once,
- what happens when a call is not answered,
- how retries are handled,
- and how outcomes are written back into your system.

### 3. RapidCall API

The workflow engine uses the RapidCall API to initiate each outbound call.

For each call, it sends the required information such as:

- agent ID,
- originating phone number,
- destination number,
- and any dynamic variables needed for the conversation.

Once the call is created, the API returns a call ID, which can be used to track the call and connect later results back to the original record.

### 4. Outcome logging

After the call ends, RapidCall sends the result back through webhooks.

These webhook events can include:

- call outcome,
- transcript,
- recording URL,
- sentiment,
- extracted data,
- and other configured fields.

Your workflow engine receives this data and writes it back to the contact list, CRM, or database. This closes the loop and keeps your records updated automatically.

### Same architecture at any scale

This four-part structure works whether you are calling 10 contacts or 10,000.

What changes at scale is not the basic architecture, but how you configure:

- timing,
- concurrency,
- retry logic,
- and data handling.

That is what lets outbound campaigns remain flexible while still fitting into the tools and workflows your business already uses.
