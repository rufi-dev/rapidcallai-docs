## Infrastructure Requirements

| Component | Minimum | Recommended |
|-----------|---------|-------------|
| **Server** | 1 vCPU, 1GB RAM | 2 vCPU, 4GB RAM |
| **Agent** | 2 vCPU, 2GB RAM | 4 vCPU, 8GB RAM |
| **PostgreSQL** | 1 vCPU, 1GB RAM | 2 vCPU, 4GB RAM |
| **Voice Agent** | 2 vCPU, 4GB RAM | 4 vCPU, 8GB RAM |

## Deployment Architecture

RapidCall AI consists of several services behind a reverse proxy: the API server, the dashboard (static), the voice agent processes, and a PostgreSQL database. All services are containerized and managed via Docker Compose.

## Security Checklist

<Warning>
Complete this checklist before going live with real phone calls.
</Warning>

- [ ] **Set strong `JWT_SECRET`** — At least 64 random characters
- [ ] **Set `AGENT_SECRET`** — Unique secret for agent-server communication
- [ ] **Enable HTTPS** — Use a reverse proxy (Nginx, Caddy) with TLS certificates
- [ ] **Restrict CORS** — Set `CORS_ORIGINS` to your domain only
- [ ] **Use environment variables** — Never hardcode secrets in code
- [ ] **Database backups** — Automated daily backups of PostgreSQL
- [ ] **Rate limiting** — Enabled by default, verify limits match your traffic
- [ ] **Firewall rules** — Only expose ports 80/443 publicly

## Reverse Proxy (Caddy)

RapidCall AI uses Caddy for automatic TLS and reverse proxying:

```
api.yourdomain.com {
    reverse_proxy localhost:8787
}

dashboard.yourdomain.com {
    reverse_proxy localhost:8080
}

crm.yourdomain.com {
    reverse_proxy localhost:8788
}
```

Caddy automatically provisions and renews TLS certificates via Let's Encrypt.

### Alternative: Nginx

```nginx
server {
    listen 443 ssl;
    server_name api.yourdomain.com;

    ssl_certificate /etc/ssl/certs/yourdomain.crt;
    ssl_certificate_key /etc/ssl/private/yourdomain.key;

    location / {
        proxy_pass http://localhost:8787;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## Scaling

### Horizontal Scaling

- **Server** — Stateless, can run multiple instances behind a load balancer
- **Agent** — Each agent process handles one call at a time. Scale by adding more processes
- **Voice infrastructure** — Supports distributed deployment with multiple nodes

### Database

- Use **connection pooling** (PgBouncer) for high concurrency
- Enable **pgvector** index for knowledge base searches
- Regular `VACUUM ANALYZE` for performance

## Monitoring

### Health Endpoints

```bash
# Server health
curl https://api.yourdomain.com/health

# Metrics (Prometheus format)
curl https://api.yourdomain.com/metrics
```

### Key Metrics to Monitor

- Server response time (p50, p95, p99)
- Active calls count
- LLM latency per call
- Database connection pool utilization
- Agent process CPU/memory usage
- Call success/failure rate

## Backup Strategy

```bash
# Daily PostgreSQL backup
pg_dump -h localhost -U rapidcall rapidcall | gzip > backup_$(date +%Y%m%d).sql.gz

# Keep 30 days of backups
find /backups -name "*.sql.gz" -mtime +30 -delete
```

## Voice Agent Deployment

For production, deploy the voice agent on dedicated infrastructure with sufficient resources for real-time audio processing. Consider multi-region deployment for global coverage and low latency.

<Tip>
Deploy the voice agent in the same region as your SIP trunks for optimal audio quality and minimal latency.
</Tip>
